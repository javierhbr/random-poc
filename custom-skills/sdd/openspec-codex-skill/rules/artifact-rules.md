# Reusable Artifact Rules

## Global rules
- Keep artifacts consistent with current repo conventions.
- Prefer incremental, reviewable change sets.
- Do not invent requirements that the user did not ask for.
- Call out unknowns explicitly instead of hiding them.

## Proposal rules
- State the problem in plain language.
- Separate goals from non-goals.
- Note rollout, rollback, and stakeholder impact when relevant.

## Specs rules
- Use concrete requirements and scenarios.
- Keep ADDED, MODIFIED, and REMOVED sections explicit.
- Focus on behavior and acceptance criteria.

## Design rules
- Explain why the chosen solution is preferred.
- Record interfaces, dependencies, and failure modes.
- Include diagrams or flows when complexity warrants it.

## Tasks rules
- Make tasks executable by another engineer or agent.
- Keep each task narrow and testable.
- Reflect dependencies between tasks clearly.
