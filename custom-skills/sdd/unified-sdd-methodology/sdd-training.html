🔥 Why This Matters

Sound familiar?

These are the patterns that slow teams down. SDD is designed to fix all of them.

🤔

"I thought we agreed on this?"

Requirements live in email threads. Without a written spec, every meeting spawns a different memory of what was decided.

🚧

Half-built features discovered late

Developers start coding before requirements are clear, then redo work when the real requirements emerge during testing.

🔥

One change breaks three things

Nobody mapped which parts of the system depend on each other, so a small change cascades into a production incident.

📋

Specs that gather dust

Documents are written, ignored during development, and outdated by the time the feature ships. Nobody trusts them.

🏃

Planning skipped to ship faster

Teams rush to code without enough clarity. They end up slower because of rework, bugs, and misaligned expectations.

🗣️

Words mean different things

"Customer", "profile", "validation" — Product and Engineering use the same words with completely different meanings.

🏠 The Big Idea

Building software is like building a house.

You wouldn't hire builders and just say "build me a house." You'd create a blueprint first. SDD is your software blueprint process.

"Imagine you hire a builder and say: 'Build me a kitchen.' Two months later, you walk in and the stove is on the wrong wall, there's no dishwasher, and the countertops are too small. Was the builder wrong? Maybe not — because nobody wrote down what you actually wanted."

The golden rule

Phases must happen in order. Skipping Specify before Plan means developers build based on assumptions. That's how you end up with the wrong product.

Phase 1 of 5
1

🏛️ Platform — Set the Foundation

Owner: Architect · Support: Product, Team Lead

Before any single feature starts, the team writes down the shared rules that everyone must follow. This is done once and updated only when the big picture changes.

It defines quality standards, who owns what parts of the system, how pieces connect, and a shared vocabulary.

🏠 House analogy: Writing the building codes, creating a map of which rooms belong to which part of the house, and agreeing what words mean ("a bedroom must have a window and a closet, not just four walls").

What gets created in Platform:

⚖️

Constitution

The explicit, testable rules every team must follow. Not vague values — specific standards like "all endpoints must return errors in this exact format."

🗺️

Ownership Map

Which team owns which service. What each service explicitly does NOT own. This turns "who's responsible?" from a debate into a simple lookup.

🔗

Dependency Map

Which services must coordinate when changed. Three tiers: must ship together / watch after deploy / no coordination needed.

📖

Shared Glossary

Every shared term defined precisely, with "what it is NOT" clauses. Prevents spec ambiguity before it starts.

⚙️

Project Config

Durable rules encoded in a reusable config that all changes inherit. Separates stable standards from per-feature details.

👥

Role Map

Who owns which phase. How decisions escalate. What "support" means versus "owner." Agreed before work begins.

Phase 2 of 5
2

🔍 Assess — Understand the Request

Owner: Team Lead · Support: Product, Architect, Manager

When a new feature request arrives, the Team Lead figures out what kind of work it is before diving in. Is it a small fix or a big change? Does it affect one service or many?

The output is a "change package" — a clearly-scoped container that says: here's what we're changing, how big it is, and who's affected.

📏 House analogy: A client says "I want to change the kitchen." You first figure out: is this just new paint (quick job), or knocking down walls (major project)? Who needs to be involved? Which plumbers and electricians must coordinate?

Questions Assess answers:

  • 🏗️ Is this a new thing (greenfield) or changing something existing (brownfield)?
  • 📐 Is this small, medium, or large?
  • 👤 Which team / component is the primary owner?
  • 🔗 Which other services are affected?
  • 📊 Does this need a full spec, or is it a quick fix?

Outputs of Assess:

  • 📦 A scoped change package with one clear owner
  • 🏷️ Size and impact classification (separately)
  • 🎯 Selected path (quick / standard / architecture-heavy)
  • 🎟️ Initial JIRA chain created (platform issue → epics)
  • ❓ Known unknowns listed explicitly
💡
Important: size ≠ impact

A one-line change can break 10 other services (small size, huge impact). Always classify them separately. The dependency map from Platform makes this a lookup, not a judgment call.

Phase 3 of 5 — Your Most Important Phase
3

✍️ Specify — Write Exactly What You Want

Owner: Product Manager · Support: Team Lead, Architect, Developers

This is where Product Managers have the most impact. You write exactly what the feature should do — in plain, testable language. Not "make it better" but "when a user enters an invalid email, show an error message within 2 seconds."

📐 House analogy: The architectural blueprint. It says: the kitchen is 4×5 meters, has an island, faces north, oven here, fridge there. Not the engineering drawings yet — but specific enough to build from.

✅ What a good spec looks like:

  • 📝proposal.md — problem + goals + non-goals
  • ➕ADDED sections — new behavior in testable sentences
  • ✏️MODIFIED sections — exactly what changes
  • 🗑️REMOVED sections — what gets taken away
  • 📖All terms defined in the shared glossary

❌ Signs a spec is too weak:

  • 🚩 "Improve the user experience"
  • 🚩 "Make the form faster"
  • 🚩 "Enhance the email validation"
  • 🚩 No non-goals listed
  • 🚩 Can't write a test from this requirement
💡
The test you should always ask

Can I write a test for this requirement? If you can't answer "yes or no" based on the spec, it's too vague. Rewrite it until you can. Define non-goals — what the feature will NOT do is just as important as what it will.

Phase 3 · Artifacts & Rationale

What Specify produces — and why.

Specify defines the "what" before anyone touches the "how." Inside a component repo, only OpenSpec is used from this point forward. No BMAD, no Speckit inside the change package.

Phase Flow
Assessed change package
scope + size + impact + path + known unknowns
↓
OpenSpec (component repo — exclusive)
proposal.md
Problem statement, goals, non-goals, acceptance summary in plain language. Separates goals from non-goals to prevent scope expansion during execution. Must be approved before Plan.
Delta specs (ADDED / MODIFIED / REMOVED)
OpenSpec-native format. Captures behavioral change explicitly — not what the system looks like, but what changes. Explicit enough for acceptance test generation.
Speckit — platform-side only, optional
Last point where Speckit is permitted. Used upstream to expose ambiguity before the component package is approved. Never mixes into the local artifact chain.
Glossary check (rule O-2)
Every term in the proposal and delta specs must appear in the shared glossary before the proposal is approved. Missing terms are added first.
🔒 Component boundary rule
Once work is inside the component repo: OpenSpec only. Mixing BMAD or Speckit creates two competing sources of intent.
↓
Spec package approved — Ready for Plan
✓ Ready to move to Plan when:
  • ✅ Goals and non-goals are explicit
  • ✅ Major behavior changes are testable sentences
  • ✅ All terms in glossary; platform-ref.yaml confirmed
  • ✅ Team agrees planning can start without guessing
⚠️ Common challenges:
  • 🚩 Spec becomes a design doc too early — move technical detail to Plan
  • 🚩 Requirements are vague or promotional — rewrite as specific behaviors
  • 🚩 Scope keeps expanding — keep non-goals visible at all times
Phase 4 of 5
4

🗺️ Plan — Map Out How to Build It

Owner: Architect · Support: Team Lead, Developers, Product

The Architect takes the approved spec and figures out how to technically build it. This creates a design document and a task list — small, verifiable steps developers can work through without guessing.

🔧 House analogy: The structural engineer, plumber, and electrician look at the blueprint and create detailed engineering drawings. They break it into tasks: "install this pipe on day 1, wire this circuit on day 2."

📄 design.md

The technical execution model. Records:

  • • Architecture and data flow decisions
  • • Service boundaries and interfaces
  • • Dependencies and rollout constraints
  • • Testing strategy
  • • ADRs — why this approach over alternatives

✅ tasks.md

The implementation work order. Each task:

  • • Maps to a JIRA story
  • • Has a clear action and target
  • • Has explicit validation criteria
  • • Is small enough to review in a single PR
  • • Has ordered dependencies (task A before task B)
ℹ️
Why Plan is separate from Specify

Specify says what to build. Plan says how to build it. Mixing them creates specs that turn into design docs prematurely — where the PM is making technical decisions and the Architect is making product decisions. Keep them clean.

Phase 4 · Artifacts & Rationale

What Plan produces — and why.

Plan converts the approved spec into a technical execution model. Every artifact produced here must trace back to the proposal and forward to the tasks developers will implement.

Phase Flow
Approved spec package
proposal + delta specs + platform refs confirmed
↓
platform-ref.yaml impact tiers → design constraints
Tier 1 must_change_together → hard constraint in design.md
Tier 2 watch_for_breakage → rollout risk in design.md
Tier 3 adapts_independently → no coordination needed
↓
OpenSpec — component repo only
design.md
Architecture decisions, service boundaries, data flow, dependencies, rollout constraints, testing strategy. Lives inside the change package — traceable to proposal, forward to tasks. A BMAD architecture doc outside the package would require manual synchronisation.
OpenSpec — component repo only
tasks.md
Implementation work order. Each task maps to a JIRA story, carries explicit validation criteria, and feeds directly into Deliver's CLI tracking. Speckit also has a tasks command — but those are standalone, not inside the change package.
OpenSpec — when needed
ADRs
Documents significant technical decisions. Component-local decisions live inside the change package. Decisions affecting shared platform truth go to the platform repo. BMAD surfaces the need for an ADR — the ADR itself is always an OpenSpec artifact.
↓
Implementation-ready plan — Ready for Deliver
✓ Ready to move to Deliver when:
  • ✅ Every major technical choice maps back to the spec
  • ✅ Tasks are executable without large hidden gaps
  • ✅ Dependencies, risks, and validation needs are visible
  • ✅ Stories or story groups are clear enough for reviewable PRs
⚠️ Common challenges:
  • 🚩 Plan drifts from the approved spec — trace every decision to a requirement
  • 🚩 Tasks too large or vague — each needs a target, action, and validation step
  • 🚩 Planning becomes architecture theater — document only what the change size justifies
Phase 5 of 5
5

🚀 Deliver — Build, Review, Ship, Close

Owner: Team Lead · Support: Developers, QA, Architect, Product

Delivery happens in small, reviewable slices — not one big dump. Each slice has its own review, verification, and sign-off before moving on.

🏗️ House analogy: Construction happens room by room. After the kitchen plumbing is done, an inspector checks it before the walls close. You don't finish the whole house and check everything at the end — that's too risky.

The 6 sub-steps inside Deliver:

1
Build — Developers implement tasks in the agreed order
2
Create PR — Open a reviewable pull request linked to the change package
3
Review PR — Code review must be resolved before merging. Not optional.
4
Verify — QA collects evidence that behavior matches the spec from Specify
5
Deploy — Team Lead coordinates release timing, dependencies, and rollback readiness
6
Archive — Close the change package. Promote the new spec as canonical truth. Required.
🔒
Archive is not optional

A feature is not done when the code is merged. It is done when the change package is archived and the spec reflects what was actually built. Skipping archive means the next person working on this feature starts from incorrect documentation.

Phase 5 · Artifacts & Rationale

What Deliver produces — and why.

Deliver is entirely OpenSpec-driven inside the component repo. The same change package that held the proposal, spec, and design now tracks execution — and is only closed when archive is complete.

Phase Flow — 6 sequential steps, all OpenSpec
🔨
Build
Implement tasks in order. Update task status in the change package as you go.
🔗
Create PR
PR references change package, tasks, and story keys. Not a freeform commit — a traceable delivery artifact.
👀
Review PR
All review feedback resolved before deploy. Review is a required gate, not a formality.
✅
Verify
Evidence co-located with the requirements it proves. Collected during delivery, not only at the end.
🚀
Deploy
Tier 1 dependents confirmed ready before merging. Rollback readiness noted.
📁
Archive
Required. Merges delta specs into main spec dir. Change package closed. New truth promoted.
All 6 steps tracked inside the same OpenSpec change package · No other tool used in the component repo
✓ Change is closed only when:
  • ✅ Planned slices are complete or intentionally deferred
  • ✅ PRs reviewed and all feedback resolved or tracked
  • ✅ Tier 1 dependency verification recorded in archive
  • ✅ Change package archived — spec reflects delivered reality
  • ✅ JIRA and platform alignment chain updated
⚠️ Common challenges:
  • 🚩 Delivery becomes one big batch — enforce slice-based execution
  • 🚩 PRs too large to review safely — align to task boundaries
  • 🚩 Archive skipped — treat it as a required closure step, not cleanup
  • 🚩 Verification happens too late — collect evidence during delivery
👟 Walk-Through Example

Real story: updating an email address.

A feature request arrives. Let's follow it through all 5 phases. Click each step to see what happens.

📬 The Request

"Customers should be able to update their email address with proper validation and clear error messages if something goes wrong."

1
🏛️ Platform — What shared rules apply?
Architect checks the foundation already built
▾

The Architect looks at the platform foundation:

  • Ownership map: "profile-service" owns email data. "auth-service" also cares — it uses email for login.
  • Dependency map: profile-service + auth-service must change together (Tier 1 — same release).
  • Glossary: "Customer" already defined. "Validation" means format check + uniqueness check.

No surprises — the foundation was set up once. The team can proceed confidently.

2
🔍 Assess — How big is this?
Team Lead scopes and classifies the work
▾
  • Type: Brownfield (existing system, adding new behavior)
  • Size: Medium (two services, multiple stories)
  • Path: Standard spec-first (not a quick fix)
  • JIRA chain created: PLAT-123 (platform issue) → PROF-456 (profile epic) → AUTH-234 (auth epic)

Now we have a change package with a clear scope and owner. No planning or coding starts yet.

3
✍️ Specify — What exactly should it do?
PM writes the spec
▾

The PM writes delta specs:

  • ADDED: Customer can submit a new email via profile settings
  • ADDED: System sends a verification email before activating the change
  • ADDED: If email already taken → "This email is associated with another account"
  • ADDED: If format invalid → "Please enter a valid email address"
  • NOT changing: Login still works with old email until verified

The proposal and spec are reviewed. Planning begins only after everyone approves.

4
🗺️ Plan — How do we build it safely?
Architect creates the blueprint
▾

The Architect writes design.md:

  • profile-service: new endpoint for email update + token storage
  • auth-service: reads new email only after verification confirmed
  • Both services must ship together (Tier 1 — from dependency map)

Then tasks.md: PROF-789, PROF-790, PROF-791 — each mapped to a JIRA story with clear acceptance criteria.

5
🚀 Deliver — Build, review, ship, archive.
Team Lead coordinates the full delivery
▾
  • Developers build task by task, updating the change package as they go
  • Each PR references the change package ID, tasks, and JIRA stories
  • Code reviewed before merge. QA verifies against the spec from Step 3.
  • profile-service and auth-service deploy together (Tier 1 constraint)
  • Archive: Change package closed. New spec becomes canonical truth.

The feature is truly done — not "code merged" done. Verified, deployed, and the spec updated.

👥 Roles

Every phase has one owner.

Clarity about ownership prevents the most common failure mode: "I thought YOU were handling that."

🏛️

Architect

Owns: Platform + Plan

Sets the technical foundation and creates the implementation blueprint. Makes sure technology choices are sound and the plan is realistic.

👩‍💼

Team Lead

Owns: Assess + Deliver

Evaluates incoming work and coordinates delivery. Sets the right scope at the start and keeps delivery on track through to archive.

📋

Product Manager

Owns: Specify

Defines exactly what to build in plain, testable language. The quality of the spec determines the quality of everything that follows.

💻

Developers

Supports: Plan + Deliver

Build from the plan, raise edge cases during Specify, update task status, write PRs linked to the spec, flag when reality differs from design.

🧪

QA / Validation

Supports: Deliver

Verifies that what was built matches what was specified. Collects evidence that the feature works as described — against the spec, not memory.

ℹ️
Support vs. Own

Every phase has one primary owner who makes the final call on whether the phase is ready to move on. "Support" means you contribute expertise and raise concerns — but you don't make the final decision.

🔧 The Tools

Three tools, one methodology.

SDD combines three AI-powered tools. Each has a specific role — they are not interchangeable.

⚖️

Speckit

The rules and quality guardian. Helps write the platform constitution — explicit, testable principles every change must follow. Surfaces hidden assumptions and ambiguities before they become expensive.

Used in:
🏛️ Platform 🔍 Assess (opt.) ✍️ Specify (platform)
📦

OpenSpec

The change package manager — the backbone of the methodology. Creates and tracks the change package from assessment to archive. Every artifact lives here: proposal, spec, design, tasks, PRs.

Used in:
🔍 Assess ✍️ Specify 🗺️ Plan 🚀 Deliver
🗂️

BMAD

The planning navigator. Classifies incoming work and picks the right depth of planning. Knows whether you're building something new or changing existing systems — and how much planning each needs.

Used in:
🏛️ Platform 🔍 Assess 🗺️ Plan (platform)

🔑 The most important rule about tools

For shared platform work

You can use all three tools together — Speckit + OpenSpec + BMAD.

Inside component repos (local work)

OpenSpec ONLY. No mixing allowed.

📌 Key Rules

Rules that prevent the most common failures.

These aren't bureaucracy — each rule exists because someone paid a high price to learn it.

🚫

No planning before Specify is approved

If the spec isn't approved, the plan has nothing to map to. Planning from a vague spec creates a plan for the wrong product.

🚫

No coding before planning is complete

Starting implementation without a plan creates surprises mid-delivery. Developers end up making architectural decisions they shouldn't be making.

📖

All terms must be in the glossary

Every word used in a spec or acceptance criterion must be defined. "Customer" to a PM might not be "customer" to a developer.

📁

Archive is not optional

A feature is not done when it's deployed. Done means verified, deployed, and the change package archived so the spec reflects reality.

🎯

Size and impact are separate

A small change can have a huge impact — one line that breaks 10 services. Always classify them separately, using the dependency map.

🔒

One owner per phase, always

When everyone is responsible, no one is. Each phase has exactly one owner who decides when the phase is ready to move on.

📅 Rolling It Out

Don't adopt everything at once.

SDD rolls out in two safe iterations. Master the front half first, then add the back half. This prevents methodology overload.

Iteration 1

Master this first — months 1 to 3

🏛️ Platform

Write the shared rules, ownership maps, and glossary once.

🔍 Assess

Turn every new request into a scoped change package.

✍️ Specify

Write clear, testable specs before any technical planning begins.

✓ Success = Teams can define shared context once and produce specs clear enough to plan without guesswork.

Iteration 2

Add this after Iteration 1 is working — months 4+

🗺️ Plan

Turn approved specs into technical blueprints and ordered tasks.

🚀 Deliver

Build in slices, review each one, verify, deploy, and archive.

✓ Success = Teams deliver work in controlled slices and never close a change without archiving it.
💡
Why two iterations?

Strong specs (Iteration 1) dramatically improve the quality of plans and delivery (Iteration 2). If you try to change planning and delivery before teams write good specs, you're fighting two problems at once. Sequence matters.

📚 Quick Reference

Words you'll hear often.

A short glossary of SDD terms. If you hear a word not in here, ask for it to be added to the shared glossary — that's the SDD way.

Change Package
The single container for everything related to one piece of work — spec, design, tasks, PRs, archive state. The "project folder" for a feature.
Delta Spec
A spec that says exactly what's ADDED, MODIFIED, or REMOVED — compared to how the system works today. Not the final state, the change.
Constitution
Platform-level document of principles every team must follow. Not suggestions — explicit and testable standards written before any work starts.
Archive
The final step of Deliver. Closing the change package and promoting the new spec as canonical truth. Not done without it.
Dependency Map
Platform document showing which services depend on each other and how urgently they coordinate. Three tiers: change together / watch / independent.
Platform vs Component
Platform = shared foundation (rules, contracts, ownership). Component = individual service or app. Platform changes affect everyone.
Impact Tier
Tier 1 = must ship together. Tier 2 = watch after deploy. Tier 3 = no coordination needed. Drives JIRA structure decisions mechanically.
Proposal
First artifact in a change package. Written by the PM. States problem, goals, and non-goals in plain language. Must be approved before planning.
Brownfield vs Greenfield
Brownfield = changing an existing system (most work). Greenfield = building something brand new. They need different planning depths.
🎯 Your First Step

You own the Specify phase.

That's where the quality of every feature is decided. A strong spec leads to the right product. A weak spec leads to rework, frustration, and the wrong product.

"Write down clearly what you want to build, agree on it before planning, plan before building, build in small reviewable steps, and never close a feature until the spec reflects what you actually shipped."
— Spec-Driven Development in one sentence

How to write your first spec — 5 steps:

1. Write the problem in 2 sentences
2. List 3 goals and 2 non-goals
3. Write one ADDED behavior as a testable sentence
4. Ask: could a developer write a test from this?
5. If yes → your spec is ready for review
