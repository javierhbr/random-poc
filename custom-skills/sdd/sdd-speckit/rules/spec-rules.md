# Spec Rules

## What belongs in the spec
- user goals
- business behavior
- constraints
- acceptance outcomes
- failure behavior
- non-goals

## What does not belong in the spec
- premature framework choices
- detailed database design unless it is a hard requirement
- implementation task lists

## Quality bar
- every requirement should be understandable by a product partner
- every major behavior should be testable
- ambiguity should be flagged for clarify
