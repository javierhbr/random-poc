The missing

Spec-Driven Development.

From historical specs to a living source of truth for what the system currently does — for humans, and for the agents that will read it.

TL;DR — six things to know
01 SDD methodologies — OpenSpec, EARS, LID — excel at change management and traceability. They do not maintain what the system currently does.
02 Over time, spec repos become historical archives. Features, proposals, deprecations, and experiments accumulate. The current operational truth is nowhere.
03 At enterprise scale — components inside platforms inside ecosystems — both humans and AI agents must reconstruct current behavior from code, ADRs, tickets, and tribal knowledge.
04 The fix is a /current-behavior layer: a continuously maintained, repository-native representation of active behavior, business rules, flows, and operational contracts.
05 Product behavior drives architecture — not the reverse. Sagas, CQRS, Redis, cloud migrations should support business behavior. The behavior layer stays stable underneath all of it.
06 Product owns intent. Engineering owns implementation. Both own the truth. Drift between what the system should do and what it does after deployment is the explicit signal this framework is designed to surface.

Chapter 02 · The Current Problem

Most spec systems become historical archives.

They contain useful information, but over time they mix too many temporal states together. When everything lives in the same bucket, it becomes hard to know what is actually true today.

  • Engineers must read many specs to reconstruct truth
  • AI agents must infer behavior from fragments
  • Deprecated ideas may still look active
  • Future proposals get confused with shipped reality
  • Impact analysis becomes slow and uncertain
  • Context retrieval becomes expensive and noisy

Chapter 03 · Where existing methods reach their edge

EARS is clear,
but not current.

EARS is excellent for writing unambiguous behavioral requirements. It reduces ambiguity, standardizes language, and helps both humans and AI understand intent.

But EARS describes required behavior. It does not tell us whether that behavior is active today, was replaced, is future work, or is partially implemented.

WHEN a card's fraud risk score exceeds the auto-lock threshold THEN the servicing platform SHALL lock the card and notify the cardholder within 30 seconds. // Clear requirement. // But — is it active today? // Or replaced by ADR-014? // Or still pending?

◈EARS · what it does well

  • Reduces ambiguity
  • Standardizes requirement language
  • Improves readability
  • Helps validation
  • Helps humans and AI understand intent

⊗EARS · what it leaves open

  • Is this behavior active today?
  • Was it replaced?
  • Is it future work?
  • Is it deprecated?
  • Is it still the operational truth?

LID draws the arrow —
at the project level.

Linked-Intent Development goes further than EARS. It treats design as the source of truth, with an explicit arrow of intent — HLD → LLDs → EARS → Tests → Code — where changes flow downstream and the docs stay current.

Inside a project, this is excellent. @spec annotations in code link back to requirements, and arrows track coherence across the codebase.

What LID intentionally scopes to the project — not by oversight, but by design — is the ecosystem layer: how dozens of LID-managed projects compose into a platform, how their current behaviors interact, how a change in one system ripples through the others.

// LID — the arrow of intent HLD → LLDs → EARS → Tests → Code @spec CARD-LOCK-001 export function lockCardOnFraud() { ... } // Strong within one project. // What about across the // platform? Across the // ecosystem? Across teams?

◈LID · what it does well

  • Treats design docs as source of truth
  • Cascades changes downstream through the arrow
  • Links code to specs via @spec annotations
  • Tracks coherence across a single project
  • Optimizes for the project over time, not just the next change

⊗What's still open at scale

  • How do multiple LID projects compose into a platform?
  • Where is the ecosystem-wide behavioral view?
  • How do agents reason about cross-project impact?
  • How is current behavior separated from change history?
  • What is the navigable graph between systems?

The missing layer this paper proposes complements LID — not replaces it. LID gives you the arrow inside a project. The behavioral source of truth and graph extend that thinking across many projects, the platform, and the broader ecosystem.

◈ Architecture / Implementation Product / Behavior ⊗
If a new engineer joins on Monday — or an AI agent opens the repo for the first time — where do they read about what the product actually does?
Why this imbalance exists

Methodologies drifted toward implementation.

Over time, our writing optimized around how the system is built — components, infrastructure, patterns, resiliency. The why and the what of the product stopped being written down. Behavior moved into tribal memory.

What ends up documented

Architecture & patterns
↓
Technical decomposition
↓
Component & infra decisions
↓
Behavior — assumed, not written

ADRs proliferate. Sagas, CQRS, Redis, microservices are described in depth. Meanwhile the product's actual behavior lives in people's heads — and disappears when they leave.

What should be documented first

Business & product behavior
↓
Operational expectations
↓
Functional contracts
↓
Architecture & implementation choices

Write behavior first — explicitly, in the repo, as a first-class artifact. Architecture documentation then explains how we chose to deliver that behavior. Both matter; only one is currently written.

The proof

Architecture can change. Behavior should not.

Cloud migrations, database swaps, adopting Redis, moving to events — these are architectural moves. If our docs are mostly about architecture, every one of these rewrites the documentation. If behavior were documented separately, it would stay stable through all of it.

Architecture · changes often written
  • High-level & low-level architecture
  • Deployment topology
  • Latency & throughput characteristics
  • Resiliency & observability patterns
  • Storage, caching, messaging choices
  • Vendor & infrastructure footprint
Behavior · should be stable rarely written
  • Business workflows
  • User expectations & experience
  • Operational contracts
  • Functional guarantees
  • Platform semantics
  • Product capabilities
The current-state behavioral source of truth gives the product side of the shelf a real home — so architecture can evolve underneath, and the product's intent stays explicitly written, not assumed.

Chapter 08 · Where traceability belongs

Tags are powerful. Where you put them matters more.

Tags like @future, @rule, @adr are useful for categorizing intent, decisions, and constraints — but they belong at the active behavior and specification layer. In the code we just need @spec.

⊗ Annotations everywhere card-controls.ts
@feature(card-lock-on-fraud) @rule(fraud-threshold-lock) @rule(notify-cardholder-on-lock) @flow(dispute-90-day-window) @flow(auto-lock-flow) @adr(pci-tokenization-strategy) @adr(card-event-contracts) @future(biometric-unlock) @governance(pci-dss-compliant) @owner(team-card-servicing) @sla(30s-lock-latency) export async function lockCardOnFraud() { // ... actual logic buried below ... }
The code becomes a governance document. Business, architectural, and operational metadata all compete with the logic. Hard to read, harder to maintain.
◈ Spec ID as the anchor card-controls.ts
@spec(SPEC-card-lock-fraud-v2) export async function lockCardOnFraud() { // implementation here. // everything else — rules, ADRs, // future capabilities, governance — // lives in the spec, not the code. }
One link. The code traces to the spec; the spec traces to everything else. Implementation stays implementation.
◈ Inside the codebase
Code traces to spec.

A single, stable anchor inside the source. Nothing else.

@spec(SPEC-card-lock-fraud-v2)
  • ↳Component implementation
  • ↳Tests reference the same spec ID
  • ↳Commits / PRs reference the spec ID

Chapter 09 · The knowledge graph

Behavior, specs, components, tests, APIs, decisions — connected.

A simple, version-controlled JSON graph. No heavy database. AI-friendly. Easy to regenerate. The graph supports two complementary views: a hierarchical drill-down from enterprise to component, and a flat network of cross-cutting relationships.

Chapter 13 · Product and Technology Collaboration

Neither product nor engineering
can maintain this layer alone.

Product understands what the business is supposed to do. Engineering understands what the system actually does at runtime. The framework stays consistent only when both sides operate against the same artifact, in the same repository, under a shared contract.

13.1 · Shared Operating Principles

Three principles both sides must agree on first.

I
One artifact, two readers.

The specification layer is a single source of truth that must be readable by a product manager during discovery and by an engineer during implementation. Neither side maintains a private fork.

II
Product owns intent. Engineering owns implementation. Both own the truth.

Product defines what should be true. Engineering ensures the specification reflects what is true after deployment. The truth of the system — what it does today — is a shared responsibility. Drift between intent and implementation is the explicit signal the framework is designed to surface.

III
Behavior outlives implementation.

When product behavior is stable, engineering is free to refactor, migrate, and re-platform without renegotiating with the business. This is the contract product and engineering exchange for the discipline of maintaining the layer.

13.2 · Role Ownership Across the Four Houses

Shared but not symmetrical.

Each house has a primary owner, a secondary contributor, and an approval gate. The framework fails when no single role is accountable for a given house.

House Primary Owner Secondary Approval Gate
/current-behavior Engineering Lead / Tech Lead Product Manager Both required before promotion
/changes Product Manager Engineering Lead, Architect Product: proposal · Engineering: design
/decisions Architect / Staff Engineer Engineering Lead, Product (business ADRs) Architect approves; Product reviews on business impact
/traceability & /graph Platform / DevEx Team All contributors Automated regen · Human review on contract breaks

Product managers do not write /current-behavior directly — engineers do. But product managers read and validate it continuously, because only product owns whether that behavior is the right behavior.

13.3 · Role Responsibilities Across the Workflow

Five stages. Explicit handoffs.

Stage Product Engineering
Discovery Authors proposal: business value, target users, success criteria Reviews feasibility; flags affected components; identifies cross-platform impact; drafts acceptance criteria
Validation Validates EARS clauses match business intent Validates technical accuracy of preconditions and triggers; confirms requirements are testable and L02/L01 impact is understood
Implementation Available for clarification; does not approve code Writes execution logic, links @spec anchors; reviews architectural conformance; builds test suites against EARS clauses
Graph regen Reviews surfaced dependency changes for business impact Resolves contract breaks before merge; reviews graph structural changes; verifies test coverage against new nodes
Promotion ◈ Approves promotion to current-behavior Executes promotion; archives change directory; records new ADRs; confirms regression suite passes
The most important handoff is the final one. Promotion requires explicit product sign-off. If product does not recognize the specification as describing the live system, the specification is by definition wrong — even if the code works.
13.4 · Communication and Cadence

Three rituals. No extra process.

Weekly · 30 min Spec triage

Product and engineering leads review /spec/changes/ together. Agree which proposals advance to validation, which need rework, and which are deferred. Replaces unstructured backlog grooming with a behavior-first conversation.

Per release · 15 min Promotion review

Before any change is promoted to current-behavior, product and engineering jointly confirm the specification matches deployed behavior. Tied to deployment cadence.

Quarterly Behavioral retrospective

Product, engineering, and architecture review the current-behavior layer at L03/L02 levels. Identify drift, deprecated behaviors still documented as active, and gaps where tribal knowledge has not been captured.

These three rituals replace a sprawling ecosystem of stand-ups, PRDs, design reviews, and tribal-knowledge transfers. The artifact is the meeting agenda.

13.5 · The Anti-Pattern

Documentation as engineering tax.

When we collectively treat the specification layer as someone else's responsibility, it quietly becomes the same historical archive the framework was designed to prevent — only with a cleaner directory structure.

⊗ Signals of breakdown
  • Product asks engineers "what does the system do?" instead of reading /spec/current-behavior
  • Engineers update code without updating the specification — and CI/CD does not catch it
  • Architecture decisions happen in chat messages or meetings without producing an ADR
◈ The remedy

The remedy is not more process. It is enforcing the contract that a change is not deployed until the specification is updated and approved by both sides.

This must be a build-pipeline gate, not a guideline.

13.6 · The Role of AI Agents

A productive third participant.

◈ Discovery

Draft specifications

Translate informal product narratives into EARS-compliant draft specifications, which product and engineering then refine.

◈ Implementation

Verify contracts

Check that proposed code changes do not break contracts declared in /current-behavior before a human reviews the PR.

◈ Promotion

Flag mismatches

Compare the draft specification against the diff being merged and surface discrepancies before product is asked to sign off.

The agent is not a replacement for either role. It lowers the cost of maintaining the layer — which is the single biggest objection product and engineering will raise when the framework is first introduced.

Chapter 14 · Long-Term Vision

From managing specifications
to operational behavioral modeling.

The long-term goal is not better documentation, better specs, or better governance in isolation. It is an operational behavioral model for AI-native enterprises. Five trajectories follow from this foundation.

01
Composable enterprise behavior

When every component publishes its active behavior, enterprise-wide operations can be assembled programmatically rather than reverse-engineered from code and organizational memory.

02
Stable behavior under shifting infrastructure

Cloud migrations, database changes, and architectural refactors become low-risk because business behavior is documented independently of implementation choices.

03
AI agents as first-class consumers

Agents read a clean, current, machine-readable behavioral graph instead of triangulating from contradictory sources — dramatically improving the safety and economy of automated changes.

04
Auditable governance

Compliance, ownership, and architectural constraints live in the specification layer, where they can be programmatically queried and audited without polluting source code.

05
Cross-platform impact analysis

The L01–L04 hierarchy and the knowledge graph make it tractable to answer "if I change this, what breaks?" across an entire multi-platform ecosystem — without reverse-engineering the organization first.

Closing · The final vision

From managing
specifications to managing operational knowledge.

Humans and AI agents should be able to quickly understand what the system currently does, why it behaves that way, how it evolved, what depends on it, and how a new change should impact it — without reverse-engineering the entire organization from code.
