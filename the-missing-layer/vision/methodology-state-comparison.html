How each methodology
manages current state

Four spec-driven methodologies — OpenSpec, LID + EARS, Spec Kit, and BMAD — each treat the evolving source of product truth differently. This analysis maps where current state lives, how change happens, and where governance gaps appear.

Focus
Behavioral source of truth over time
Running Example
Card Balance · 3 spec changes
Verdict
No single methodology is sufficient alone
Recommendation
Layered hybrid model
Summary

TL;DR — at a glance

Each methodology has a distinct answer for where current product state lives. Only OpenSpec has a native, first-class model for controlled evolution over time.

Methodology Where "current state" lives How change happens Strength Gap / Risk Model
OpenSpec openspec/specs/ as current agreed behavior Proposed changes as deltas → archived/merged into specs Best at preserving evolving product behavior as source of truth Needs discipline to keep implementation aligned Native
LID + EARS Intent graph: HLD → LLD → EARS claims → tests → code Change cascades down the graph from product intent to code Best traceability from intent to code Needs a change-governance process around the graph Native-ish
Spec Kit Feature folders + constitution.md + specs/plans/tasks Sequential flow: specify → plan → tasks → implement Best for structured feature execution with AI agents No native long-term canonical product-state model Custom layer
BMAD PRD, architecture, stories, sprint status, project context Agile phases: analysis → planning → solutioning → implementation Best role-based product/tech collaboration workflow Documents drift unless explicitly reconciled Distributed
The current state of a product should not live only in code, tickets, or ADRs. It should live in a behavior-first specification that product and technology both understand.
— Core argument · The Missing Layer Working Paper
Running Example

Card Balance · three spec changes

Each methodology is evaluated against the same feature: a payment card balance view that evolves through three discrete specification changes. This shows how each methodology absorbs and tracks those changes.

Live Feature · Card Balance
Initial State

Change 1

Methodology 1 of 4

OpenSpec
behavioral source of truth

★ Strongest for state governance

OpenSpec is the clearest native model for current product state over time. Its core insight is that current behavior lives in the main specs, while future or proposed work lives in separate change folders. This gives you a first-class distinction between accepted behavior, proposed deltas, and historical completed changes.

The lifecycle

01
Current specs openspec/specs/ — accepted product behavior
02
Proposed change / delta openspec/changes/<id>/ — behavior delta
03
Implementation Code changes against the delta
04
Archive change Delta merged back into specs
05
New current state openspec/specs/ describes the new behavior

  specs/
    card-balance/
      spec.md  ← CURRENT STATE

  changes/
    add-pending-transactions/
      proposal.md
      design.md
      tasks.md
      specs/card-balance/
        spec.md  ← delta
    add-balance-freshness/
      specs/card-balance/spec.md
    add-negative-balance-warning/
      specs/card-balance/spec.md

  archive/
    2026-05-16-add-pending-transactions/
    2026-05-16-add-balance-freshness/
    2026-05-16-add-negative-balance-warning/

Current state file (after 3 changes)

# Card Balance Specification ## Requirements ### Requirement: View card balance The system SHALL allow an authenticated customer to view the current balance and available balance for an active card. ### Requirement: Show pending transaction total ← Change 1 The system SHALL display the total pending transaction amount. ### Requirement: Show stale balance indicator ← Change 2 The system SHALL indicate when balance data is stale. Data is stale when the last refresh is older than 15 minutes. ### Requirement: Show negative balance warning ← Change 3 If current balance is below $0, the system SHALL show a negative balance warning.

A change delta (Change 1)

openspec/changes/add-pending-transactions/specs/card-balance/spec.md ## ADDED Requirements ### Requirement: Show pending transaction total The system SHALL display the total pending transaction amount. #### Scenario: Customer views balance with pending transactions Given the customer is authenticated And the customer has an active card And the card has pending transactions When the customer opens the card balance view Then the system SHALL display the pending transaction total.
Take: OpenSpec is the strongest of the four if your priority is: "Product behavior must remain the canonical source of truth, and implementation should follow it." It has a native first-class model for proposed vs. accepted vs. archived behavior.

✓ Strengths

  • Native distinction: current vs. delta vs. archived
  • Specs describe behavior, not implementation
  • Verifiable behavior contracts, not code details
  • Clear lifecycle for evolving product behavior
  • Historical change traceability via archive

⊗ Gaps

  • Needs discipline to keep implementation aligned
  • No built-in role collaboration model
  • Does not give you code-level traceability (@spec)
  • Change governance must be added explicitly
Methodology 2 of 4

LID + EARS
traceability graph

★ Best for spec-to-code traceability

LID — Linked-Intent Development — manages state as a traceable graph of intent. Code files are reachable from tests, tests from specs, specs from low-level designs, and low-level designs from the high-level design. EARS requirements provide atomic, structured, grep-addressable behavioral claims.

The intent chain

Product / Domain
HLD — why the product exists and what behavior matters
Design Intent
LLD — component-level design intent
Behavioral Claims
EARS — atomic, structured requirement IDs
Implementation Proof
Tests + Code @spec annotations
Change propagates top-down: update HLD/LLD intent, add/modify EARS IDs, update tests referencing those IDs, update code with @spec annotations.

### CB-BAL-001 through CB-BAL-003
... (unchanged)

### CB-BAL-004: View pending transaction total

When an authenticated customer opens
the card balance view for an active card
with pending transactions, the system shall
display the total pending transaction amount.

### CB-BAL-005: Show stale balance indicator

If the last successful balance refresh is

// Code annotation — @spec links implementation back to EARS ID // @spec CB-BAL-004 <p>Pending transactions: {balance.pendingTransactionTotal}</p> // @spec CB-BAL-005 {balance.isStale && <Alert>Balance may be outdated</Alert>} // @spec CB-BAL-006 {balance.currentBalance < 0 && ( <Alert>Your balance is below zero</Alert> )}
Take: LID is very strong for traceability. It answers: "Where is this behavior defined, tested, and implemented?" But by itself, LID is not as strong at managing the lifecycle of proposed vs. accepted product behavior. Use it under a higher-level spec/change model like OpenSpec.

✓ Strengths

  • EARS IDs are atomic, grep-addressable claims
  • @spec annotations link code back to requirements
  • Full chain: HLD → LLD → EARS → tests → code
  • EARS patterns cover all behavioral modes
  • Makes hidden behavior visible via traceability

⊗ Gaps

  • No native "current vs. proposed vs. archived" model
  • Needs governance process around the graph
  • More engineering-centric; product ownership unclear
  • Current state is a graph, not one canonical file
Methodology 3 of 4

Spec Kit
feature execution scaffold

★ Best for AI-agent feature execution

Spec Kit is more of a feature execution scaffold for spec-driven development than a long-term product-state registry. It creates a constitution for principles, and structured feature folders for each discrete piece of work. The missing layer: there is no native canonical current-state model without adding one.

The execution flow

/specify
Define what and why Motivations and functional requirements, no technical decisions
/plan
Define how Frameworks, libraries, databases, infrastructure
/tasks
Break into steps Manageable chunks respecting dependencies
/implement
Execute tasks Validate constitution + spec + plan + tasks exist, then execute

Constitution example

# Product Constitution — .specify/memory/constitution.md ## Principle 1: Product Behavior First Product behavior SHALL be defined before implementation details. ## Principle 2: Balance Data Transparency Any customer-facing balance view SHALL show when the data was last updated. ## Principle 3: Traceable Requirements Every implemented behavior SHALL trace back to a feature specification and acceptance criteria. ## Principle 4: No Silent Financial Risk If a balance state may negatively affect the customer, the system SHALL clearly communicate it.
The gap: Spec Kit does not have a native "current accepted behavior vs. pending delta vs. archived change" model. Current state is reconstructed from completed feature specs — unless you add a governed product-state/ layer explicitly.

Recommended: generated current state file

1. Completed feature specs are marked "accepted" 2. A script or agent scans accepted specs 3. It extracts functional requirements + acceptance criteria 4. It writes/updates product-state/card-balance.md 5. Product-state becomes the readable current behavior file npm run generate:product-state -- card-balance Read: specs/001-card-balance/spec.md specs/002-card-balance-pending-transactions/spec.md specs/003-card-balance-freshness/spec.md specs/004-card-balance-negative-warning/spec.md Generate: product-state/card-balance.md

✓ Strengths

  • Excellent for turning idea → structured execution
  • AI-agent native: commands scaffold complete flows
  • Constitution provides stable project principles
  • Clear separation: specify (what) → plan (how) → tasks
  • Especially useful for greenfield or feature-level work

⊗ Gaps

  • No native canonical product-state model
  • Current state must be reconstructed from feature specs
  • No native "proposed vs. accepted vs. archived" model
  • Needs additional layer for long-term state governance
Methodology 4 of 4

BMAD
role-based collaboration workflow

★ Best for product–tech collaboration

BMAD manages current product state through role-based, agile-style documents and workflows. It excels at explaining how Product, PM, Architect, UX, Dev, QA, and Technical Writer can work together. But product state is distributed across PRD, addenda, decision logs, architecture, stories, sprint status, and project context — and it can drift unless explicitly reconciled.

The four phases

I
Analysis Product brief, research, problem definition
II
Planning PRD, addenda, decision logs, change proposals
III
Solutioning Architecture, ADRs, epics, stories
IV
Implementation Sprint status, code, tests, retrospectives

The gap: BMAD's product state is distributed across PRD, addenda, decision logs, architecture, stories, sprint status, and project context. That is useful, but documents can drift unless you define which artifact is canonical and add a reconciliation step.
# Recommended BMAD reconciliation step 1. Product updates PRD or PRD addendum 2. Architect updates architecture or ADRs 3. PO/SM creates epics and stories 4. Dev implements stories 5. QA verifies acceptance criteria 6. Reconciliation step updates: docs/current-state/card-balance.md

✓ Strengths

  • Best model for role-based collaboration
  • Each phase produces documents for the next
  • Named roles: Analyst, PM, Architect, Dev, QA, UX
  • ADRs capture architectural decisions with rationale
  • project-context.md acts as a constitution-like file

⊗ Gaps

  • Product state is distributed across many documents
  • No native "current vs. proposed vs. archived" model
  • Documents drift without explicit reconciliation step
  • Canonical current behavior unclear without a custom file

Synthesis

The recommended
combined model

No single methodology covers all dimensions. The best model combines them in a layered architecture where each plays to its strength.

Layer 01 · Collaboration

BMAD

Explains how product and technical roles collaborate. Provides the workflow from brief → PRD → architecture → stories → implementation.

Product brief · PRD · Architecture · ADRs · Epics/stories
Layer 02 · Canonical Behavior

OpenSpec

Owns the current accepted product behavior. Distinguishes current specs from proposed deltas and archived completed changes.

Current behavior specs · Proposed deltas · Archived changes
Layer 03 · Traceability

LID + EARS

Connects behavior to tests and code. EARS IDs are atomic claims; @spec annotations prove the implementation still matches intent.

Requirement IDs · Tests linked to claims · Code @spec links
Layer 04 · Execution

Spec Kit

Accelerates feature execution. Turns a spec into a structured plan, task list, and implementation scaffold — especially with AI agents.

Specify · Clarify · Plan · Tasks · Implement
